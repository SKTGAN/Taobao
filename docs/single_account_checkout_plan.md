# 单账号定时下单第一阶段方案

## 目标

第一阶段只支持一个已扫码登录的淘宝账号和一个商品任务。程序先进入确认订单页，用户在可见 Chrome 中核对商品、SKU、数量、收货地址、价格、号码保护和协议选项，再明确授权任务；程序在计划时间只提交已经核对的订单，订单创建成功后停在淘宝官方待付款页面，不读取支付信息，也不自动支付。

## 安全边界

- 不保存淘宝密码、Cookie 内容、支付账号、支付密码或支付密钥。
- 不处理或绕过验证码、滑块、短信验证和账号保护页面。
- 不实现动态代理池、Cookie/UA 轮换、浏览器指纹伪装或模拟行为规避风控。
- 出现验证页、登录失效、价格异常、SKU 未选择或页面结构无法识别时，任务暂停并记录原因。
- 自动提交订单前必须由用户显式勾选授权；授权只对当前任务和当前计划时间有效。

## 第一阶段状态机

```text
草稿
  -> 预检中
  -> 待授权
  -> 款式预检中
  -> 待核对订单
  -> 授权检查中
  -> 已武装
  -> 等待中
  -> 触发中
  -> 提交中
  -> 待付款
```

任何运行状态都可以进入：

- `需人工处理`：登录、验证码、SKU、库存、地址或价格需要用户确认。
- `失败`：浏览器不可用、页面结构无法识别、超时或重试耗尽。
- `已取消`：用户主动停止任务。

## 分步实现

1. 浏览器会话：稳定复用单账号 Chrome，并能在应用重启后读取 `DevToolsActivePort` 重新连接。
2. 页面控制：通过本机 Chrome DevTools Protocol 读取当前页面状态和执行有限的点击，不导出 Cookie。
3. 预检：确认登录状态、商品域名和购买按钮；授权时由用户填写本次 SKU ID 或粘贴同一商品的已选款式链接，并确认款式说明与数量。
4. 款式复检：程序仅保留稳定的商品 ID 与 SKU ID，切换准备好的商品页并再次确认购买按钮可用，然后提前进入确认订单页，但不提交订单。
5. 订单核对：用户人工处理地址、价格、数量、号码保护和协议选项，关闭隐私/协议/规则说明标签后执行最终授权。
6. 定时器：使用系统时间确定目标时刻，使用单调时钟完成最后等待；保持已人工核对的确认订单页可用。
7. 下单流程：到点只点击已核对确认订单页中的“提交订单”；若确认页失效或出现隐私/协议辅助标签，任务安全停止，不会重新点击购买绕过核对。
8. 结果识别：检测待付款/支付页面并写入任务与订单记录；验证码或异常页转为 `需人工处理`。
9. 回归测试：先在本地模拟商城验证状态机和点击逻辑，再用低价普通商品进行人工监护测试。

## 第一阶段验收标准

- 应用重启后，已打开的账号 Chrome 不需要全部关闭即可重新连接。
- 未授权任务不能自动提交订单。
- 计划时间支持毫秒，并能在页面已预热的前提下按时触发。
- 失败原因能在运行日志中定位到登录、浏览器、页面、库存、SKU、提交或验证阶段。
- 模拟商城测试能完整走完“商品页 -> 确认订单页 -> 待付款页”。
- 真实平台只运行到待付款，不触发任何支付动作。
